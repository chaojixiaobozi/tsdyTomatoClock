# AGENTS — 协作说明

本文件描述**长期稳定**的协作基线：项目背景、核心能力、技术选型、编码规范。  
**功能范围、验收标准、版本迭代**以 `docs/PRD.md` 为准；**实现方式、状态机与接口级约定**以 `docs/TECH_SPEC.md` 为准。二者随版本更新，**不必**为同步它们而频繁改本文件。

---

## 1. 项目背景

- **产品**：运行在 macOS 上的本地番茄钟，帮助用户按「专注 → 短休 →（周期性）长休」节奏工作。
- **形态**：原生桌面体验（窗口、通知、系统集成）；数据以**本地**为主，不依赖账号与云端（除非 PRD 另行定义）。
- **取向**：默认少配置即可用；界面简洁、动效克制；明确「不做」的范围见 PRD（协作、复杂项目、强制登录等）。

---

## 2. 核心功能点（能力级）

以下按**能力**归纳，不绑定某一版 PRD 小节编号；具体规则与交互以 PRD 为准。

| 能力 | 说明 |
|------|------|
| **多阶段计时** | 工作 / 短休 / 长休等阶段可配置时长；展示剩余时间与当前阶段；开始、暂停、继续、跳过、重置等控制。 |
| **轮次逻辑** | 按配置在工作与休息之间切换，并支持「若干次工作后进入长休」等规则。 |
| **到点提醒** | 系统通知等弱提醒为主；若 PRD 要求更强感知（如窗口前置、音效），在 ViewModel / Service 实现，**不**塞进 Domain。 |
| **阶段衔接策略** | 阶段之间是自动进入还是需用户确认、跳过如何计数等，**一律以 PRD + TECH_SPEC 当前条文为准**，实现与单测与之对齐。 |
| **时长预设** | 常见节奏一键应用（具体名称与分钟数以 PRD 表为准）。 |
| **极简记录** | 如当日完成数等本地统计，持久化策略见 TECH_SPEC。 |

## 3. 框架选型

| 维度 | 约定 |
|------|------|
| 语言 / UI | **Swift 5**、**SwiftUI**；最低系统与 Deployment Target 与 TECH_SPEC 一致（当前 macOS 12+）。 |
| 计时模型 | **结束时刻 + 暂停偏移**（单调时间），避免仅靠 Timer 累加导致可见漂移。 |
| 持久化 / 通知 | **UserDefaults**（或 TECH 约定的等价方案）；**UserNotifications** 做本地通知。 |
| 结构 | **Domain**（纯 Swift、零 SwiftUI） / **ViewModel** / **Services** / **SwiftUI 视图**；系统窗口、激活、文件等放边界层。 |
| 质量 | **XCTest**（逻辑）+ **XCUITest**（UI）；合并前 CI 应能通过 TECH 约定的测试命令。 |
| 视觉 | 番茄色系、低饱和、长时观看友好；动效克制；浅色 / 深色跟随系统。 |
| 工程 | Xcode 工程与脚本路径以仓库内实际目录为准（如 `TomatoClock.xcodeproj`、`scripts/`）。 |

## 4. 编码规范

1. **分层**：状态机、剩余时间、轮次与计数等**必须**在 **Domain**；Domain **禁止** import SwiftUI / AppKit。窗口前置、通知、持久化由 **ViewModel 调用 Service** 完成。
2. **可测性**：Domain 行为变更须带 **单元测试**；时间相关逻辑通过入参 `Date` 或可注入的时钟抽象测试，避免在 Domain 内写死「当前时刻」导致不可测。
3. **UI 与无障碍**：关键控件设置 **`accessibilityIdentifier`**；主要流程尽量键盘可达；少把断言绑死在易变的文案上。
4. **改动纪律**：只改与任务相关的文件，不做顺手大范围重构；命名、文件组织与现有代码保持一致。
5. **体验与安全**：破坏性操作（如重置）需确认；权限被拒（如通知）时界面有清晰说明。默认**不**上传用户行为数据；本地存什么、键名含义在代码或 TECH 中可追踪即可。
6. **文档与实现一致**：若 PRD / TECH 已更新而代码未跟上，优先在 PR 中说明范围；避免「文档一套、实现另一套」长期并存。
